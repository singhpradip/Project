# Boardstack — Hands-On Build Guide (Learn by Doing)

**Build the whole stack by hand, one command at a time, understanding every piece.** This is the teaching companion to `SETUP.md` (which is the terse reference). Here, each step follows the same rhythm:

> **What** you'll run · **Why** it exists · **What just happened** · **✅ Verify**

Take it slowly. Don't paste whole sections — run one block, read the explanation, confirm the "Verify" check, then move on. If something breaks, jump to **§13 Troubleshooting** — the errors listed there are ones already hit on this exact project.

Your machine already has **Node v24** and **npm 11**, which is perfect (anything ≥ 22.18 works). Related docs: `boardstack-system-design.md` (architecture), `boardstack-data-model.md` (entities), `PROGRESS.md` (task tracker).

## 0. The mental model (read once before typing)

You are building **one repository** that contains **three JavaScript packages** and talks to **one database**:

```
boardstack/                 ← the monorepo (one git repo)
├── boardstack-web/         ← Next.js frontend (the browser UI)
├── boardstack-api/         ← Express backend (business logic + the ONLY thing that touches the DB)
├── packages/shared/        ← Zod schemas shared by web + api (validation in one place)
└── docker-compose.yml      ← runs PostgreSQL (and later api/web) in containers
```

Three ideas to hold in your head the whole way through:

1. **Separation of concerns.** The browser never talks to the database. The web app calls the API over HTTP; only the API opens database connections. This is what makes the app secure and testable.
2. **One source of truth for validation.** The rules for "what a valid issue looks like" live once, in `packages/shared`, and are imported by *both* the form (frontend) and the request handler (backend).
3. **The database enforces tenant isolation itself** (via Row-Level Security), so no coding mistake can leak one company's data to another. We'll set this up deliberately.

Keep the big picture in mind and each command below will feel like it has a purpose, not just magic.

**What**

```bash
mkdir -p packages/shared/src/schemas

```

Edit `packages/shared/package.json` so it reads (keep the version npm generated):

```json
{
  "name": "@boardstack/shared",
  "version": "1.0.0",
  "type": "module",
  "main": "src/index.ts",
  "types": "src/index.ts",
  "exports": { ".": "./src/index.ts" },
  "dependencies": { "zod": "^3.23.0" },
  "devDependencies": { "typescript": "^5.6.0" }
}
```

Create `packages/shared/src/index.ts`:

```ts
export * from "./schemas/issue";
```

Create `packages/shared/src/schemas/issue.ts`:

```ts
import { z } from "zod";

// An "enum" here is a fixed set of allowed string values.
export const issueTypeEnum = z.enum(["task", "bug", "story", "epic"]);
export const priorityEnum = z.enum(["low", "medium", "high", "urgent"]);

// A Zod "schema" is a runtime description of a valid object.
export const createIssueSchema = z.object({
  title: z.string().min(1).max(200),
  description: z.string().max(20_000).optional(),
  type: issueTypeEnum.default("task"),
  priority: priorityEnum.default("medium"),
  assigneeId: z.string().uuid().nullable().optional(),
  storyPoints: z.number().int().min(0).max(100).nullable().optional(),
  labelIds: z.array(z.string().uuid()).default([]),
  dueDate: z.coerce.date().nullable().optional(),
});

// z.infer turns the runtime schema into a compile-time TypeScript type — for free.
export type CreateIssueInput = z.infer<typeof createIssueSchema>;
```

```bash
cd ../..   # back to repo root
```

**Why.** **Zod** is a validation library: you describe the shape of data once, and it can both *check* data at runtime (throwing helpful errors) and *generate the TypeScript type* at compile time. By putting these schemas in a shared package, the browser form and the server handler validate against the exact same rules — impossible for them to drift apart.

- `"type": "module"` — use modern ES module `import`/`export` syntax.
- `"exports"` — declares the public entry point so other packages import `@boardstack/shared` cleanly.

**What just happened.** You created your first real package. `z.infer<typeof createIssueSchema>` is the magic moment: `CreateIssueInput` is a TypeScript type derived automatically from the validator, so type and validation can never disagree.

**✅ Verify**

```bash
ls packages/shared/src/schemas    # issue.ts present
```

## 4. PostgreSQL in Docker (+ the two-role security model)

**What.** Create `docker-compose.yml` at the repo root:

```yaml
services:
  db:
    image: postgres:16
    container_name: boardstack-db
    restart: unless-stopped
    environment:
      POSTGRES_DB: boardstack
      POSTGRES_USER: postgres
      POSTGRES_PASSWORD: postgres
    ports:
      - "5432:5432"
    volumes:
      - pgdata:/var/lib/postgresql/data

volumes:
  pgdata:
```

Start it:

```bash
docker compose up -d db
docker compose ps          # STATUS should show "Up"
```

Create the **application role** (a second database user):

```bash
docker exec -it boardstack-db psql -U postgres -d boardstack -c "
  CREATE ROLE boardstack_app WITH LOGIN PASSWORD 'app_password' NOBYPASSRLS;
  GRANT CONNECT ON DATABASE boardstack TO boardstack_app;
  GRANT USAGE, CREATE ON SCHEMA public TO boardstack_app;
"
```

**Why.** **Docker** runs Postgres in an isolated container so you don't install a database on your Mac directly — and everyone who clones the repo gets the identical database with one command. **docker-compose** describes containers declaratively.

- `image: postgres:16` — the official Postgres image, version 16.
- `ports: "5432:5432"` — expose the container's Postgres port to your Mac so tools can connect to `localhost:5432`.
- `volumes: pgdata:...` — persist the data on a named volume so it survives container restarts. (`docker compose down -v` deletes this volume; plain `down` keeps it.)

The **two-role model** is the heart of the security story:

| Role | Power | Used by |
|---|---|---|
| `postgres` (owner) | superuser — can create tables, bypass security | migrations only |
| `boardstack_app` | `NOBYPASSRLS`, no superuser | the running API |

The app connects as `boardstack_app`, a role that is *physically incapable* of bypassing Row-Level Security (§6.4). So even a buggy query can't read another tenant's rows. Migrations, which need to create tables, use the powerful `postgres` owner.

**What just happened.** Postgres is now running with a `boardstack` database, and you created a deliberately weak `boardstack_app` login the API will use at runtime.

**✅ Verify**

```bash
docker exec -it boardstack-db psql -U postgres -d boardstack -c "\du"
# You should see both 'postgres' and 'boardstack_app' in the roles list.
```


**Why.** **Express** is a minimal web framework: you register middleware and route handlers, and it turns incoming HTTP requests into function calls.

- **The two URLs** in `.env` are the whole point of §4: `DATABASE_URL` uses the weak `boardstack_app` role (runtime), `DIRECT_URL` uses the `postgres` owner (migrations). We'll wire each to the right place in §6.

## 6. Prisma + the database schema

Prisma is your **ORM** — it lets you define tables in a schema file and query them with a fully typed JavaScript client, instead of writing raw SQL by hand.

### 6.1 Install and initialize

```bash
# in boardstack-api
npm install --save-dev prisma @types/pg
npm install @prisma/client @prisma/adapter-pg pg
npx prisma init --datasource-provider postgresql
```

**Why each package:** `prisma` (the CLI for migrations/generation), `@prisma/client` (the query library), `@prisma/adapter-pg` + `pg` (the driver that actually connects to Postgres), `@types/pg` (its TypeScript types).

`prisma init --datasource-provider postgresql` creates `prisma/schema.prisma`, a `prisma.config.ts`, and appended env entries. The `--datasource-provider` flag pre-sets Postgres so you don't hand-edit it.

### 6.2 Point the CLI at the OWNER connection (important!)

Open `boardstack-api/prisma.config.ts`. Change the datasource URL from `DATABASE_URL` to `DIRECT_URL`:

```ts
  datasource: {
    url: process.env["DIRECT_URL"],   // ← was DATABASE_URL
  },
```

**Why this matters (this is the exact bug we hit).** `prisma migrate` needs to create a temporary "shadow database" to detect schema drift — that requires owner privileges. If the CLI uses `DATABASE_URL` (the weak `boardstack_app` role) you get `Error: P1010: User was denied access`. Pointing the *CLI* at `DIRECT_URL` (the `postgres` owner) fixes it. Your *app* will still use the weak role — different connection, set in §6.5.

### 6.3 Define your first models

Open `boardstack-api/prisma/schema.prisma`. The `generator` and `datasource` blocks are already there. Add these models (start small — the full catalog is in `boardstack-data-model.md`):

```prisma
model Organization {
  id          String       @id @default(uuid()) @db.Uuid
  auth0OrgId  String       @unique @map("auth0_org_id")
  name        String
  slug        String       @unique
  createdAt   DateTime     @default(now()) @map("created_at")
  memberships Membership[]
  projects    Project[]
  @@map("organization")
}

model AppUser {
  id          String       @id @default(uuid()) @db.Uuid
  auth0UserId String       @unique @map("auth0_user_id")
  email       String
  name        String?
  createdAt   DateTime     @default(now()) @map("created_at")
  memberships Membership[]
  @@map("app_user")
}

model Membership {
  id             String       @id @default(uuid()) @db.Uuid
  organizationId String       @map("organization_id") @db.Uuid
  userId         String       @map("user_id") @db.Uuid
  role           String
  createdAt      DateTime     @default(now()) @map("created_at")
  organization   Organization @relation(fields: [organizationId], references: [id], onDelete: Cascade)
  user           AppUser      @relation(fields: [userId], references: [id], onDelete: Cascade)
  @@unique([organizationId, userId])
  @@map("membership")
}

model Project {
  id             String       @id @default(uuid()) @db.Uuid
  organizationId String       @map("organization_id") @db.Uuid
  key            String
  name           String
  createdAt      DateTime     @default(now()) @map("created_at")
  organization   Organization @relation(fields: [organizationId], references: [id], onDelete: Cascade)
  @@unique([organizationId, key])
  @@map("project")
}
```

**Why the annotations:**
- `@id @default(uuid())` — primary key that defaults to a random UUID.
- `@map("...")` / `@@map("...")` — Prisma uses camelCase in code but maps to snake_case column/table names in the DB (our SQL convention).
- `@relation(...)` — declares a foreign key; `onDelete: Cascade` means deleting an org deletes its memberships/projects.
- `@@unique([...])` — a composite uniqueness rule (a user has one membership per org; a project key is unique per org).

### 6.4 Create the tables (first migration)

```bash
npx prisma migrate dev --name init
```

**What just happened.** Prisma compared your schema to the (empty) database, generated a `migration.sql`, and applied it — creating the `organization`, `app_user`, `membership`, and `project` tables. It also generated the typed client. (If it says "Already in sync, no schema change," your models weren't saved — re-check the file.)

### 6.5 Instantiate the client with the app role

Create `boardstack-api/src/lib/prisma.ts`:

```ts
import "dotenv/config";
import { PrismaPg } from "@prisma/adapter-pg";
import { PrismaClient } from "../generated/prisma/client.js";

// Runtime connects as the weak app role (RLS applies). Note the import path:
// Prisma 7 generates the client into src/generated/prisma, NOT "@prisma/client".
const adapter = new PrismaPg({ connectionString: process.env.DATABASE_URL });
export const prisma = new PrismaClient({ adapter });
```

**Why the odd import path (another bug we pre-empted).** Prisma 7's new generator outputs the client to `src/generated/prisma/` (check `output` in `schema.prisma`). Importing from `@prisma/client` would fail. The adapter is fed `DATABASE_URL` — so every query the app makes runs as `boardstack_app`, under RLS.

### 6.6 Add Row-Level Security (raw SQL — Prisma won't do this for you)

```bash
npx prisma migrate dev --create-only --name rls_policies
```

This makes an **empty** migration file. Open the new `prisma/migrations/<timestamp>_rls_policies/migration.sql` and paste:

```sql
-- Turn on RLS and add the isolation policy for each tenant table.
ALTER TABLE membership ENABLE ROW LEVEL SECURITY;
ALTER TABLE membership FORCE  ROW LEVEL SECURITY;
ALTER TABLE project    ENABLE ROW LEVEL SECURITY;
ALTER TABLE project    FORCE  ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation ON membership
  USING      (organization_id = current_setting('app.current_org')::uuid)
  WITH CHECK (organization_id = current_setting('app.current_org')::uuid);

CREATE POLICY tenant_isolation ON project
  USING      (organization_id = current_setting('app.current_org')::uuid)
  WITH CHECK (organization_id = current_setting('app.current_org')::uuid);

-- The app role can touch tables, but RLS still filters the rows it sees.
GRANT SELECT, INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public TO boardstack_app;
ALTER DEFAULT PRIVILEGES IN SCHEMA public
  GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO boardstack_app;
```

Apply it:

```bash
npx prisma migrate dev
```

**Why.** **Row-Level Security (RLS)** is a Postgres feature that attaches a filter to a table so a query only sees rows matching a condition. Our condition is `organization_id = current_setting('app.current_org')` — a per-connection variable the API sets on every request (§6.7). `FORCE` makes the rule apply even to the table owner. The `GRANT`s give `boardstack_app` permission to run queries at all (RLS then restricts *which rows*). This is why cross-tenant leakage is impossible even if you forget a `WHERE` clause.

### 6.7 The tenant middleware (sets the org per request)

Create `boardstack-api/src/middleware/with-tenant-db.ts`:

```ts
import { prisma } from "../lib/prisma.js";

// Runs each request inside a transaction that sets the current org,
// so RLS scopes every query to that tenant.
export async function withTenantDb(req: any, _res: any, next: any) {
  try {
    await prisma.$transaction(async (tx) => {
      await tx.$executeRawUnsafe(
        `SET LOCAL app.current_org = '${req.tenant.organizationId}'`,
      );
      req.db = tx;         // handlers must use req.db, never the global prisma
      await next();
    });
  } catch (err) {
    next(err);
  }
}
```

**Why `SET LOCAL` inside a transaction.** Database connections are pooled and reused across requests. `SET LOCAL` scopes the variable to *this transaction only*, so tenant A's org id can never linger on the connection when tenant B's request reuses it. This is the runtime half of isolation; RLS (§6.6) is the enforcement half.

**Why.** **Next.js** is the React framework. `create-next-app` is its official scaffolder; the flags choose: TypeScript, Tailwind (utility CSS), ESLint (linting), the **App Router** (file-based routing under `app/`), a `src/` directory, and the `@/*` import alias (so `@/components/...` means `src/components/...`). The added libraries each own one job (this separation is a rule in `CLAUDE.md`):

| Library | Owns |
|---|---|
| **React Query** | server data — fetching, caching, optimistic updates |
| **Zustand** | ephemeral UI state — open modals, active filters, drag state |
| **React Hook Form** + `@hookform/resolvers` + Zod | form state + validation (reusing the shared schema) |
| **@auth0/nextjs-auth0** | login session with Auth0 |

Now wire React Query in. Create `boardstack-web/src/app/providers.tsx`:

```tsx
"use client";
import { QueryClient, QueryClientProvider } from "@tanstack/react-query";
import { useState } from "react";

export function Providers({ children }: { children: React.ReactNode }) {
  const [client] = useState(() => new QueryClient());
  return <QueryClientProvider client={client}>{children}</QueryClientProvider>;
}
```

Wrap it in `boardstack-web/src/app/layout.tsx` (add the import and wrap `{children}` in `<Providers>`):

```tsx
import { Providers } from "./providers";
// ...inside <body>:  <Providers>{children}</Providers>
```

**Why `"use client"`.** In the App Router, components render on the server by default. React Query uses browser state (hooks), so its provider must be a **client component** — that's what the `"use client"` directive marks.


## 10. Concept deep-dives (the "why" behind the whole thing)

**Multi-tenancy.** Every company is a "tenant." All tenants share one database and one set of tables; each tenant-scoped row carries an `organization_id`. Three layers keep them apart: the Auth0 token says which org you're in, the API middleware checks your membership, and Postgres RLS filters rows at the database. Defense in depth — any one layer failing doesn't leak data.

**Why RLS instead of just `WHERE organization_id = ?`.** Because humans forget `WHERE` clauses. RLS moves the guarantee from "every developer remembers every time" to "the database enforces it on every query, always." That's a far stronger promise and a great thing to be able to explain in an interview.

**The request lifecycle (once you build handlers).**

```
Browser form → React Hook Form validates with @boardstack/shared (Zod)
   → HTTP call to Express (:4000)
   → authenticate (Auth0 JWT) → resolveTenant → authorize → validate (same Zod)
   → withTenantDb: BEGIN; SET LOCAL app.current_org = <org>
   → service does the work via req.db (Prisma, app role)
   → RLS confines every row to this org
   → COMMIT → response → React Query updates the UI
```

**State ownership on the frontend.** Server data lives in React Query (never copied into Zustand). UI-only state (which modal is open) lives in Zustand. Form state lives in React Hook Form. Shareable state (filters) lives in the URL. Keeping these lanes separate prevents the classic bug of stale duplicated server data.

## 13. Troubleshooting (errors already seen on this project)

**`P1010: User was denied access on the database`** — `prisma.config.ts` is using `DATABASE_URL` (the weak app role) for migrations. Fix: point its datasource at `DIRECT_URL` (§6.2). The app role can't create Prisma's shadow database.

**`Cannot find module '@prisma/client'`** (or a client import error) — Prisma 7 generates the client to `src/generated/prisma`. Import `PrismaClient` from `../generated/prisma/client.js`, not `@prisma/client` (§6.5).

**`error: unknown option '--base-color'`** (shadcn) — that flag was removed. Use `npx shadcn@latest init -d` for defaults (§8).

**`prisma migrate dev` says "Already in sync, no schema change"** — your models weren't actually saved to `schema.prisma`, so there's nothing to migrate. Re-check the file has your `model` blocks, then re-run.

**`npm warn workspaces ... no workspace folder present`** — harmless; appears if you run an npm command referencing a workspace a moment before it exists. Ignore.

**Web can't import `@boardstack/shared`** — run `npm install` at the repo root (not inside a package) so workspaces get linked (§9).
